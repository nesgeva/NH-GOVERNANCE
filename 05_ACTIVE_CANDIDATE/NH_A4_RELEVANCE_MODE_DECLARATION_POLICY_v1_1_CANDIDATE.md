# 0. PURPOSE AND SCOPE

**What A4 is.** A4 governs how each consuming component of Attention and
Relevance Control (§7R) declares what "relevant" means for its task. This
candidate records **Ness's Option C structure decision** — one shared
relevance language, plus a mandatory component-specific declaration — and
defines the required declaration fields in plain policy form.

**What this candidate settles:** the **structure** of A4 only.

**What this candidate does not do:** it does not write any component's
actual declaration; it does not invent relevance-mode names, gate
conditions, or dimensions beyond what the governing files already settle;
it does not redesign the settled §7R contract, producers, validation, or
record schemas; it creates no B1, B26, or B-INT-1 mechanics; it integrates
nothing and adopts nothing.

# 1. NORMALIZED DECISION RECORD — NESS'S OPTION C

**Ness-approved decision (carried in the task instruction; preserved):**
A4 uses **Option C — shared list + component-specific declaration**, with
this meaning:

> - N.H has one shared relevance language / shared relevance-mode list.
> - Each component must declare which relevance mode it uses for a task.
> - Each declaration must state why that mode is being used.
> - Each declaration must state what the component does when relevance is
>   uncertain.
> - No component may invent a private hidden meaning of "relevant."
> - Relevance remains purpose-scoped only.
> - Relevance is never truth, evidence strength, causation, authority, or
>   Ness's judgment.

**Normalized statements (no new meaning added):**
1. There is exactly **one shared relevance language** in N.H; every
   component's notion of "relevant" is expressed **in that language**,
   never in a private or hidden one.
2. Using the shared language happens through a **mandatory, explicit,
   per-component declaration** — a component without a valid declaration
   has no relevance mode.
3. Every declaration carries its **reason** and its **uncertainty
   behavior**; silence on either makes the declaration invalid.
4. Relevance stays **purpose-scoped only** and is **never** truth,
   evidence strength, causation, authority, permission to act, or Ness's
   judgment.

# 3. THE COMPONENT DECLARATION — REQUIRED FIELDS (PLAIN POLICY FORM)

Every consuming component, for every relevance-using task, must have a
declaration carrying **all** of the following. Field names below are plain
policy language; their mechanical representation binds later to the
settled §7R Tier 1/Tier 2 contract and record schemas (B-INT-1), which
this candidate does not redesign.

1. **Component name / id** — which component this declaration belongs to,
   stably identified, with the declaration's own version.
2. **Task or purpose** — the settled controlled purpose type (one of the
   five-value vocabulary), plus the optional explanatory label (which
   carries no system behavior). An unrecognized purpose type follows the
   settled halt-and-propose rule; it is never guessed around.
3. **Selected relevance mode** — the declared mode (id + version) under
   the settled two-tier contract: the Tier 1 declaration validated by
   Attention and Relevance Control, and the Tier 2 rules owned and
   validated by the consuming component.
4. **Reason for that mode** — a plain, recorded statement of why this mode
   fits this task. A declaration without a reason is invalid.
5. **Uncertainty behavior** — what the component does when relevance is
   uncertain: how it treats gate-excluded candidates, unresolved or failed
   dimensions, and producer/validator disagreement — consistent with the
   settled validation outcomes and, wherever mouth-produced dimensions are
   declared, carrying the identifier and version of the Tier 2 handling
   rule as §7R already requires. Uncertainty is recorded, never silently
   smoothed over.
6. **Allowed retrieval/use boundary** — what the mode may reach and use
   for this purpose: eligible source scope and use limits, always inside
   §7Q's prior authorization (§4) and inside the consuming component's
   settled boundaries. The declaration never widens access; it can only
   narrow use.
7. **Logging / audit requirement** — the declaration and every evaluation
   under it are auditable under §0B through the settled append-only
   relevance-event and disagreement records; declarations themselves are
   versioned, append-only, never silently edited; mode changes go through
   the settled proposal-and-confirmation path with consequence surfacing.
8. **Fail-closed behavior when relevance cannot be safely determined** —
   the governed honest path (§5): stop or proceed explicitly degraded per
   the settled rules of the owning component, mark the outcome
   (degraded / context-limited / insufficient) honestly, and never
   silently guess, never silently lower a threshold, never substitute
   unrelated material, and never pretend the evaluation succeeded.

A declaration missing any field is **not a valid declaration**, and a
component without a valid declaration for the current purpose has **no
relevance mode to run** — it follows §5's fail-closed path rather than
improvising one.

# 4. PRESERVED RULE — RELEVANCE NEVER OVERRIDES ACCESS (§7Q BEFORE §7R)

- The settled ordering stands, unchanged and unbypassable: **§7Q's
  purpose-specific authorization runs before Attention and Relevance
  Control receives any candidates.** For visible surfacing, §7Q
  visible-output eligibility applies; for internal use, §7Q internal-use
  authorization applies; §7R operates **only on already-authorized
  material** and never owns, reruns, or redefines §7Q.
- On the visible side, the settled final-gate order (accepted B24 §6.5,
  consumed as-is) stands: **§7Q first, SACL second** — a relevance
  judgment never substitutes for, reorders, or weakens either gate.
- **A relevance declaration can never widen access.** No mode, dimension,
  gate condition, reason, or uncertainty rule makes ineligible material
  eligible, reveals withheld material, or signals that withheld material
  exists.

---

# 5. PRESERVED RULE — NO PRETEND SUCCESS, NO SILENT GUESSING

When relevance cannot be safely determined — an unrecognized purpose type,
a failed or unresolved evaluation, missing or invalid declaration, or a
producer/validator disagreement the declared rules cannot resolve — the
component follows the governed honest path:

- **say so** through the settled records (relevance-event / disagreement
  records; §0B one-operation/one-log);
- **halt or proceed explicitly degraded** according to the owning
  component's settled rules — e.g., Context Retrieval's settled
  distinction between a genuine empty result (a normal condition:
  proceed bare, marked context-limited/revisable) and a system failure
  (never claimed as success), with the fuller retrieval system-failure
  path belonging to **B26** and not designed here;
- **mark the outcome** degraded / context-limited / insufficient per the
  governed path — never a silent guess, never a silently lowered
  threshold, never substituted unrelated material, never a pretended
  success.

Nothing in this section invents new failure mechanics: it binds A4
declarations to the honest-failure rules the governing files already fix,
and leaves every open failure-path mechanic where the map places it.

---

# 6. RELEVANCE IS NOT — SEPARATIONS PRESERVED

Carried binding from Master §7R, unchanged: a relevance judgment
determines **relevance to the current purpose only**. It is **not truth,
not evidence strength, not causation, not authority, not permission to
act, not Ness's judgment, and not permanent.** It resolves no clash,
merges no Person-Boxes, rewrites/reorders/suppresses no root, reading,
telling, or clash, and writes nothing directly into the Living State Web
(relevance events are emitted, never written directly; the
relevance→currentness→relevance circular chain remains prohibited).
Positional/semantic channel identity remains retrieval provenance, not a
relevance dimension. No component may invent a private hidden meaning of
"relevant," and no evaluation collapses into a single hidden score.

# 8. HOW A4 FEEDS B1, B26, AND B-INT-1 LATER (POLICY DIRECTION ONLY)

- **B-INT-1** consumes completed A4 declarations: it instantiates each one
  against the already-settled §7R Tier 1/Tier 2 contract, validation
  flow, and record schemas, and wires the consumers — an
  implementation-neutral instantiation that does not redesign the
  contract. A4 supplies the *what*; B-INT-1 performs the *wiring*, later.
- **B1** later supplies the empirical per-mode parameter and ceiling
  *values* for retrieval modes whose declarations A4 governs; a
  declaration names its boundary and behavior, B1 tunes the numbers —
  none are chosen here.
- **B26** later designs the retrieval system-failure path; A4's
  fail-closed field (§3.8) must remain consistent with it and defers to
  it — this candidate constrains neither its mechanics nor its values.

No B1, B26, or B-INT-1 mechanical content is created, changed, or biased
