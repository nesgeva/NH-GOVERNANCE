lets the already-valid, already-settled manual reread trigger pass B10's
RR2 when accepted A25 truthfully produces **no** assignment — a typed

## 5. THE MANUAL COMPATIBILITY RECORD — TYPED CARRIER

**The record:** `manual_reread_compatibility_record` [proposed] — one
append-only, immutable record per qualifying reread claim; the second
permitted referent of B10's existing `reread_mode_ref` at RR2. It
carries **no mode**, because no A25 mode truthfully exists for this
path; it carries the settled manual basis instead.

**Record shape [all field names proposed — mechanical carriage only]:**

| Field | Content |
|---|---|
| `manual_compatibility_id` | The record's own stable identity — the value `reread_mode_ref` carries for this path |
| `record_kind` | The fixed literal `manual_no_new_information_compatibility`; any other value is malformed (§9) |
| `reread_claim_ref` | The existing B10 claim (`reread_claim_key` binding {root, trigger}) this record is bound to |
| `trigger_record_ref` | The existing §7H/B10 trigger record; its `trigger_type` MUST be `manual` |
| `settled_basis` | The fixed literal `master_v10_7H_manual` — the recorded settled authority for this record type's existence; a citation, never restated or new policy |
| `compatibility_reason` | A reference to the manual trigger record's own recorded reason (§7H requires one; the manual request itself is it) — reference, never a copy |
| `committed_at` | Timestamp |
| `schema_version` | Record schema version |

**No `assigned_modes` field exists in this record type — structurally
absent.** Nothing can be stored in a field that does not exist: no
`manual`, no `none`, no `not_applicable`, no sentinel, no empty list.
The accepted assignment record's `assigned_modes` remains exactly the
three valid values and is never touched by this package.

**Unambiguous distinction from an A25 assignment record — by
construction, on four independent grounds:**

1. **Distinct record types** with distinct required fields — the
   compatibility record has no `assigned_modes`, no `mode_evidence`, and
   no `assignment_reason`; the accepted assignment record has no
   `record_kind`, no `settled_basis`, and no `compatibility_reason`.
   The accepted record's shape is consumed exactly as accepted and is
   **not** retrofitted, extended, or modified in any way.
2. **Distinct identity keys** derived with different type strings
   (§7's `manual_compat_v1` vs the accepted `mode_assignment_v1`) —
   the two identities can never collide.
3. **The fixed `record_kind` literal** inside the compatibility record —
   an internal integrity field making any cross-labeled or forged record
   detectably malformed.
4. **A trigger-truthfulness check at consumption** (§6): a compatibility
   record is valid only for a `manual` trigger; an assignment record is
   valid only where A25 truthfully produced an assignment.

**Applicability rule — the truthfulness gate:** a compatibility record
may be committed only when **(a)** the referenced trigger record's
`trigger_type` is `manual`, and **(b)** no A25 relationship truthfully
applies for that trigger — the trigger carries no new-information
evidence bearing directly or context-materially on the target reading.
If a manual trigger **does** carry new information for which accepted
A25 truthfully produces an assignment, the accepted three-case
connection applies unchanged and a compatibility commit is **refused**
(`compatibility_not_applicable`). Determining whether an A25
relationship applies remains the open producer's territory (unchanged,
unbiased); this rule consumes the truthful result and manufactures
nothing in either direction — neither a mode where none exists, nor a
"no relationship" where one exists.

## 6. CONSUMPTION AT RR2 — HOW `reread_mode_ref` REFERENCES IT

- RR2 proceeds only from **exactly one committed mode-slot record per
  claim** — either a committed A25 assignment record (the accepted
  three-case path, byte-identically unchanged) **or** a committed manual
  compatibility record (this path). `reread_mode_ref` carries that one
  record's identity; the context snapshot's committed state references
  it, binding snapshot to record.
- The referent's type is unambiguous from its identity and shape (§5);
  RR2 additionally verifies **kind-vs-trigger truthfulness**: a
  compatibility record consumed for a non-manual trigger fails closed
  (§9), and a claim observed with **both** record types committed fails
  closed as contradictory (§9) — never auto-picked, merged, or resolved.
- Nothing in the accepted connection's three-case consumption is
  altered, widened, renamed, or re-ordered by this package.

## 7. BINDING, TIMING, AND ONE-COMMIT IDENTITY

- **Bound to exactly one manual trigger and exactly one claim:** every
  compatibility record references exactly one committed B10 reread claim
  and its one manual trigger record. No claim → no compatibility record.
- **Committed before the RR2 context snapshot:** RR2 proceeds only from
  a committed record; the commit is one atomic compare-and-commit.
- **One canonical compatibility record per claim:**
  `manual_compatibility_key = stable_hash(reread_claim_key +
  "manual_compat_v1")` [proposed] — exactly one committed compatibility
  record can ever exist for a claim; a repeat commit attempt with
  identical content converges idempotently on the existing record; a
  commit attempt with **different** content is refused and surfaced as
  conflicting evidence (§9) — never silently replaced, merged, or
  overwritten.
- **Cross-type exclusivity at commit:** the compare step refuses a
  compatibility commit when a committed A25 assignment record already
  exists for the claim (recorded refusal, `mode_slot_already_assigned`).
  The accepted assignment path's own commit rules are consumed exactly
  as accepted and are not modified; any residual both-types state is
  detected at consumption and fails closed (§6, §9).
- **Duplicate prevention and concurrency:** the single
  compare-and-commit admits exactly one winner per key; concurrent
  losers observe the committed record and append nothing.
- **Append-only and immutable:** once committed, the record is never
  edited, deleted, or recalculated for that claim. A genuinely different
  situation is a **new trigger and a new claim** under B10's existing
  rules — never a mutation of this one.

## 9. FAIL-CLOSED CONSUMPTION (EXISTING B10 VOCABULARY ONLY)

B10's existing duplicate prevention, claim identity, append-only
layering, and crash recovery are preserved unchanged. Consumption is
fail-closed using **B10's existing outcomes**, with per-condition
recorded reason codes [proposed — record-shape refinement only, no new
terminal]:

| Condition | Behavior (existing B10 vocabulary) | Recorded reason [proposed] |
|---|---|---|
| **Missing** (manual no-new-information claim with no committed mode-slot record of either type) | Exactly the current behavior — **boundary state** `dependency_blocked_held` at RR2: nothing proceeds, no mode invented, `thread_membership_v1` not borrowed. **Parent terminal (RR5): `reread_blocked_held`** when the operation truthfully terminates; while held-resumable, RR5 is not yet written | `compatibility_missing` |
| **Unreadable** (record exists; store state cannot be read) | Fail closed — **recovery state** `indeterminate_recovery_required`; nothing reconstructed. **RR5: `reread_indeterminate`** on truthful termination | `compatibility_unreadable` |
| **Malformed** (shape violates §5: wrong or absent `record_kind` literal, a missing required field, any mode-carrying field present, unbound references) | Fail closed — `indeterminate_recovery_required`; the invalid record stays visible as recorded evidence, never repaired in place. **RR5: `reread_indeterminate`** on truthful termination | `compatibility_malformed` |
| **Wrong trigger** (record bound to a trigger whose `trigger_type` is not `manual`) | Fail closed — `indeterminate_recovery_required`; no reinterpretation. **RR5: `reread_indeterminate`** on truthful termination | `compatibility_wrong_trigger` |
| **Not applicable** (the manual trigger carries new information for which A25 truthfully produces an assignment) | Commit refused (§5 gate); a residual committed record in this state fails closed — `indeterminate_recovery_required`. **RR5: `reread_indeterminate`** on truthful termination | `compatibility_not_applicable` |
| **Duplicated** (more than one committed compatibility record observed for one claim — an integrity breach of the one-commit key) | Fail closed — `indeterminate_recovery_required`; neither record is auto-picked or deleted. **RR5: `reread_indeterminate`** on truthful termination | `compatibility_duplicated` |
| **Contradictory** (a compatibility record and an A25 assignment record both committed for one claim; or a differing commit attempt against the committed record; or content contradicting the trigger record) | Exactly B10 §7 row 15's pattern — `indeterminate_recovery_required`, fail closed; **the conflict is surfaced and recorded, never resolved by the system**. **RR5: `reread_indeterminate`** on truthful termination | `mode_slot_conflicting_records` / `compatibility_contradictory` |

Every such stop emits B10's existing `fail_closed_event` §0B record with
the reason code — a **child/recovery log, never the parent terminal.**
**No new terminal type is invented and no false-success result exists.**
Every real B10 operation still receives exactly one existing RR5 parent
terminal when it truthfully reaches a terminal state
(`reread_blocked_held` for the held boundary; `reread_indeterminate` for
the recovery-required conditions), durably appended after the children
resolve, and **acknowledgement never occurs before that durable
terminal.** Resumable conditions leave RR5 unwritten until truthful
resolution — success is never falsely acknowledged.

## 10. RECOVERY (LOOKUP-FIRST; REUSE, NEVER RECALCULATE)

- **Crash before the compatibility commit:** nothing partial can exist
  (single atomic commit); recovery finds the claim in its pre-RR2 state
  (B10 §7 row 2, mode slot not permitting). The record must still be
  committed before RR2; if none is available, the claim is held
  (`compatibility_missing`) — a resumable state; RR5 is not written
  until the operation truthfully resolves.
- **Crash after the commit, before the context snapshot:** recovery
  resumes RR2 under the claim (B10 §7 row 2, mode slot now permitting)
  and **reuses the committed record exactly** — the committed record
  governs this claim; a genuinely changed situation is a new trigger and
  a new claim.
- **An existing record found on recovery:** it **is** the record for
  that claim — reused as-is; a repeat identical commit converges
  idempotently; no second record; no silent recalculation.
- **Conflicting evidence on recovery:** the §9 contradictory path —
  `indeterminate_recovery_required`, fail closed, conflict surfaced and
  recorded append-only, never auto-resolved; RR5 `reread_indeterminate`
  on truthful termination, durably recorded before any acknowledgement.
- **Duplicate recovery runs:** lookup-first no-ops per B10 §7 row 16 —
  repeats re-read committed state and re-apply nothing.

certainty. Records link by identities only; §7Q governs access to every
record, including logs of refusals. No silent internal operation exists
on this path.
