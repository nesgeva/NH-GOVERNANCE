# 2. TARGET FIRST-CLASS TELLING-RECORD CONTRACT

This section defines the **required semantics** of a first-class telling record. It chooses no file format, serialization, storage engine, or threshold.

**Two ownership classes — the DUMB/SMART boundary on the card.** Card fields fall into two ownership classes (detailed in the §2.0 location matrix and the §2.1 SMART/DUMB matrix); the boundary between them is load-bearing:

- **Semantic fields (SMART; mouth/engine-proposed; grounded by the Reading Proposal Acceptance Check):** see the SMART/DUMB ownership matrix below.
- **System/mechanical fields (DUMB; system-generated):** see the matrix below.

### 2.0 Field ownership-and-location matrix (Correction 2 — canonical ownership vs protected recovery copies)

Every value this package touches has exactly one **canonical home** — the record that *owns* it. The manifest additionally holds **protected immutable recovery copies** of payloads it must be able to reconstruct; those copies are not a second canonical home and never compete for authority (§2.0 note below). "Inside the card" means the immutable first-class telling-card payload only. The four canonical homes are: **(C)** immutable card payload, **(M)** manifest / operation metadata, **(E)** append-only event, **(R)** parent reading.

**(C) Canonical immutable first-class telling-card payload** — fixed once, never edited; includes, where applicable:
`telling_id`; `parent_reading_id`; the exact mechanically-resolved `supporting_root_ids`; `root_speaker` (source-carried — §2.1); `subject`; `perspective_owner`; optional `attribution_path`; `evidence_relationship`; `telling`; optional `stance`; optional `firmness`; **`firmness_evidence_basis` (required when `firmness` is present)**; optional `source_event_time`; **`legacy_whose` (when the embedded entry contained `whose`)**; **`legacy_theme` (when the embedded entry contained `theme`)**; **`embedded_when` (when the embedded entry contained `when`)**; the **complete applicable `derivation_provenance`** (entirely DUMB — §2.1); `embedded_entry_correspondence`; `created_at`; `produced_by`; `schema_version`; and the **record-level `telling_idempotency_key`** (the card's canonical record identity). *(Omitting `legacy_whose`, `legacy_theme`, or `embedded_when` would contradict §4A's no-loss mapping — they are canonical card fields.)*

**(M) Manifest / operation metadata — canonical home for these, plus protected recovery copies of payloads owned elsewhere:**
*Owned by the manifest:* `preparation_id`; `preparation_idempotency_key`; a separate **producing-pass/reading-operation reference** that ties the manifest to its producing pass (this is operation metadata — **not** the reading-record `idempotency_key` field); **`telling_creation_operation_key`** (the wider creation-operation identity, fixed in the manifest and used by commit/recovery/logging — *not* the card's record-level idempotency field); the expected telling count and ordered discriminator inventory; the payload and envelope integrity digests; and the protected recovery copies and references required for recovery.
*Owned by the append-only event/checkpoint records, NOT by the manifest (Correction 3):* each event/checkpoint record **canonically owns its own** identity, idempotency key, timestamp, result type/state, parent/preparation/card references, integrity-or-inventory digest (where applicable), and schema/version provenance — specifically the **manifest-validity event** (its identity, idempotency key, and `prepared_at`), **each card-commit event** (its identity, idempotency key, and `committed_at`), and the **telling-completion checkpoint / zero-telling sentinel** (its identity, idempotency key, timestamp, result type, and verified inventory/digest). The manifest may hold a **protected reference or deterministic derivation input** for these records, but it is **not** their canonical owner (§4.4.1 contracts).
*Held as protected recovery copies (canonical home is elsewhere):* the exact parent-reading payload **including the reading's own `idempotency_key`** (canonical home: the reading record — R), the exact complete telling-card payloads (canonical home: the cards — C), the exact embedded `story_layer`, and all assigned identities required for recovery.

**(E) Append-only event values — NOT immutable card payload:**
`prepared_at` (manifest-validity event only); `committed_at` (telling-card commit event only); the `abandoned`/`failed` lifecycle-event timestamps (their respective lifecycle events only).

**(R) Parent reading** — owned by the existing authoritative twelve-field reading contract (`id · reads · meaning · confidence · role · story_layer · mode · timestamp · produced_by · schema_version · derived_from · idempotency_key`), reached via `parent_reading_id`. **All twelve fields — including the reading's `idempotency_key` and the two-slot `confidence` — are canonically owned by the immutable reading record.** They **may be copied exactly into the protected manifest for recovery, but their canonical home remains the reading record**, and the manifest holds only a recovery copy. This package does not redesign or reclassify that contract. The reading's two-slot `confidence` remains reading-level metadata governed by the reading contract, is included exactly in the protected complete reading payload, is **never copied onto the telling card**, and remains separate from telling `firmness` (A2 does not reclassify its ownership).

**Note — the manifest is not a second authority.** The `prepared_reading_telling_envelope`'s recovery copies do **not** become a second semantic authority, a canonical card, a canonical reading, or an evidential vote (§9). They exist only to reconstruct the exact accepted output during recovery.

### 2.1 SMART/DUMB ownership matrix

| Field | Owner | Rule |
|---|---|---|
| `root_speaker` | **DUMB / source-carried** | Mechanically copied from the source root's authoritative `role` ("who spoke, from the root" — Master reading schema). The mouth does **not** generate, rewrite, normalize, or override it, and `root_speaker` is **never** in a mouth-produced semantic field list. |
| parent-reading `role` | **DUMB / source-carried** | Carried from the target root under the existing reading contract; **never** listed as mouth-produced or accepted semantic parent-reading content. The twelve-field reading schema is not redesigned. |
| `subject`, `perspective_owner`, optional `attribution_path` | **SMART** | The genuinely interpretive perspective roles. Proposed by the mouth; grounded by the acceptance check. Nested/reported perspectives are represented here plus via `evidence_relationship`. |
| `evidence_relationship`, `telling`, optional `stance`, optional `firmness`/`firmness_evidence_basis`, optional `source_event_time` | **SMART** | Proposed by the mouth; grounded by the acceptance check. |
| **support selection via evidence handles** (which supplied evidence supports the telling) | **SMART** | The mouth selects/claims support using **deterministic pass-local evidence handles supplied with the pass** — handles that identify only evidence actually supplied to that pass. The mouth **never invents persistent raw record IDs**. The support claim is grounded by the acceptance check. |
| `legacy_whose`, `legacy_theme`, `embedded_when` (preserved legacy telling values) | **SMART value, DUMB copy** | Their **original populated values are accepted semantic/legacy values** emitted in the embedded `story_layer` (so they passed acceptance as part of the embedded entry). **After acceptance, the DUMB layer copies them exactly** into the card's corresponding legacy fields. The copy **does not reinterpret, normalize, confirm, or enrich** them, and **omission remains omission** (an absent embedded value yields an absent card field, never an invented one). |
| resolved `supporting_root_ids` | **DUMB** | The system **resolves accepted handles to immutable root IDs**, validates **existence, pass-availability, and consistency with the reading's `reads`**. Resolution **may not select new evidence or alter the accepted support claim**. |
| `produced_by` | **DUMB** | Reproducibility-grade producer provenance consistent with the reading contract: producer/model/digest/engine/prompt/config. This is the single canonical home for model/config provenance. |
| `derivation_provenance` (entire field) | **DUMB** | The mechanical **lifecycle and lineage** facts only: lifecycle type (fresh pass / reread / materialization), the actual `pass_id`, recovery lineage, the materialized embedded-entry source (when a materialization), and the execution-path identity. It **may reference `produced_by`** but **must not duplicate competing model/config values** that could diverge. Its canonical home is the card (§2.0 C). **No "semantic provenance" sub-part exists** — none of this is interpretive. |
| `telling_id`, `parent_reading_id`, `embedded_entry_correspondence`, `created_at`, `schema_version`, all operation/idempotency identities, all integrity digests | **DUMB** | System-generated machine-state/provenance facts; assigned mechanically after acceptance (§4.3); never mouth claims. |

This realigns the card with the Master's DUMB/SMART law: *the DUMB machinery may establish directly-verifiable machine-state and provenance facts; the SMART machinery may interpret, connect, lean — but may never silently promote an interpretation into an established fact* (Master §0 S13 sharpening; Decision Defaults §2). **Only genuinely interpretive content belongs to SMART and must pass acceptance.** **No semantic field may be added, inferred, normalized, replaced, or enriched after acceptance**; mechanical assembly after acceptance attaches DUMB fields (and the exact DUMB copies of accepted legacy values) only, and is never semantic enrichment.

**Identity (allocated at preparation — prior round: id/time fix; this round: never-reused-if-abandoned)**
- `telling_id` — permanent, stable, **system-generated (DUMB), unique.** **Assigned exactly once during mechanical preparation (§4.3 step 3), after semantic acceptance and before manifest durability — never at card-commit time and never by the mouth.** It becomes **reserved and immutable when the manifest is durably valid.** Card commit uses that exact assigned id; **recovery never mints a replacement id.** **An assigned `telling_id` is never reused, even if its preparation is later abandoned** (a recorded id is permanently spent; an abandoned preparation does not return its id to circulation). It is the durable link target every downstream component points to. It is **not** the parent reading id plus a list index. The same `telling_id` is **system-wide unique across quarantine, production, recovery, and every lifecycle location**; lifecycle change never mints a new id and never retires it (§5). **Storage location is never part of the identity** — the same telling in quarantine and (if ever promoted) in production is one identity, never two records competing for canonical status. A reserved-but-not-yet-committed identity inside the manifest is **not yet a semantically usable telling card**, and **does not become a valid canonical telling merely because it was assigned** — canonical status still requires a committed, integrity-valid card and the §5A completion gate (§4, §5A).

**Required provenance links (mandatory; a telling with a broken or absent chain is not a valid telling)**
- `parent_reading_id` — the id of the immutable reading whose engine pass produced this telling. Required, single-valued. It is the **DUMB system-assigned `reading_id`** (§2.1), fixed during mechanical assembly (§4.3), so the telling's parent link is known before either record is committed.
- `supporting_root_ids` — **the exact mechanically-resolved root id(s)** that ground this telling, consistent with the parent reading's `reads`. **DUMB-resolved (§2.1):** the mouth selects/claims support using **deterministic pass-local evidence handles** supplied with the pass (handles identifying only evidence actually supplied to that pass), and that support claim is grounded by the acceptance check (§3); the DUMB layer then **resolves the accepted handles to immutable root IDs**, **validates existence, pass-availability, and consistency with the reading's `reads`** (the reading writer already "verifies each exists in the sealed roots before commit" — Master reading schema). The **mouth never invents persistent raw record IDs**, and resolution **never selects new evidence or alters the accepted support claim.** The required identity relationship is defined here; no storage format or code API is chosen. The telling points to roots by id; never copies root text; never accepted solely because an earlier reading stated it (Master §7K grounding rule, line 1702).
- `derivation_provenance` — **entirely DUMB; mechanical lifecycle and lineage only (§2.1):** the actual **lifecycle type** (fresh pass / reread / materialization), the actual `pass_id` and any recovery lineage, the **materialized embedded-entry source** when this card is a materialization, and the **execution-path identity**. It **may reference `produced_by`** for producer/model/config provenance but **does not duplicate competing model/config values** (`produced_by` is the single canonical home for those). None of this is interpretive, so **no "semantic provenance" sub-part exists.** Its canonical home is the card payload (§2.0 C).
- `embedded_entry_correspondence` — the exact, deterministic identifier of the embedded `story_layer` entry this card corresponds to: the `parent_reading_id` plus the stable per-telling discriminator (the embedded entry's ordinal index within its reading's `story_layer` list as emitted by the producing pass). The correspondence anchor used by the §3 integrity invariant. The discriminator is a correspondence/creation device only; **never** the telling's identity (that is `telling_id`).

**Structured perspective (Master §7K STRUCTURED PERSPECTIVE MODEL, lines 2192–2198 — roles kept separate, never collapsed)**
- `root_speaker` — who produced the source root. **DUMB / source-carried: mechanically copied from the source root's authoritative `role`** ("who spoke, from the root" — Master reading schema). The mouth does **not** generate, rewrite, normalize, or override it (§2.1). Nested or reported perspectives are represented through `perspective_owner`, `subject`, `attribution_path`, and `evidence_relationship`, never by altering `root_speaker`.
- `subject` — who or what the telling is about.
- `perspective_owner` — whose viewpoint, belief, feeling, or framing the telling claims to represent.
- `attribution_path` — optional; used only when perspectives are nested (e.g. `friend → quoted by father → reported by Ness`). Populated only when the root supports the chain. Never invented. Absent when not supported — honest non-presence, never a filled placeholder.

These roles may refer to the same person or to different people. The record never assumes the root speaker is automatically the subject or the perspective owner.

**Evidence relationship (mandatory for every valid first-class telling — Master §7K line 2198)**
- `evidence_relationship` — one of: direct self-report, direct quotation, reported speech, observation, or engine inference. Required on every valid first-class telling. Reported speech is not direct access to the reported person's internal state. *(This package preserves the five named categories exactly; it invents no scale, weighting, or threshold over them — A-policy where evidential weight is judged.)*

**Telling content**
- `telling` — the interpretive content, recorded unaltered as the producing pass emitted it. Immutable once written.
- `stance` — preserved as its own value when the producing pass emitted a `stance`; **not** assumed to be derivable from the general `telling` text (§4A). Present only when emitted; omitted when not, never `"unknown"`.

**Firmness (present only when supported — Master §7K FIRMNESS RULE, line 2204)**
- `firmness` — how strongly the perspective owner appears to hold the stance. Present only when the root supports it; **omitted** otherwise (never `"unknown"`, never forced).
- `firmness_evidence_basis` — when `firmness` is present, the observable signals it rests on. Provisional and revisable. Separate dimension from model confidence; never presented as direct access to a person's internal state. **The firmness scale, labels, thresholds, and scoring method are not invented here — open (Master §7K).** *(§4A governs a legacy embedded `firmness` lacking a recorded basis.)*

**Time (non-circular meanings; five distinct concepts, none conflated, none self-referential)**
- `created_at` — the telling-record's **logical creation timestamp**, assigned **once during mechanical preparation, before the payload digest is finalized**, and **immutable inside the eventual card payload**. It is a DUMB system field (above), not a mouth claim. It is **not** defined as equal to `prepared_at` and is **not** a claim about when durability completed.
- `prepared_at` — the timestamp of the **append-only manifest-validity/completion event** that proves the complete preparation became durable. It lives **in that completion event, not inside the card payload**, so it is never self-referentially embedded as proof of its own future durability. It is **not required to equal `created_at`**. (Corrects v1.3, which set `created_at = prepared_at` and described `prepared_at` as the exact durable-completion time while also placing it inside the payload before that completion had occurred — a self-reference.)
- `committed_at` — the timestamp of the **append-only first-class-card commit event**, recorded in that event, not inside the immutable card payload, and not confused with the logical creation time.
- `embedded_when` — the exact preserved legacy embedded `when` value, when the embedded entry had one. Distinct from all of the above. Omitted when absent.
- `source_event_time` — the source-event time the telling is about, when the root/source supports it. Distinct from all of the above. Omitted when unsupported; never invented.

**No exact durable-completion time is ever claimed before durable completion has occurred:** `prepared_at` is written by the completion event after the fact; `created_at` (assigned earlier, inside the payload) is a logical creation marker, not a durability claim.

**Producer and schema provenance**
- `produced_by` — reproducibility-grade provenance consistent with the reading schema's `produced_by` model (Master §6B line 543): `origin` from `{observed, imported, simulated, generated, reaction, human_annotation}`, plus, for engine-produced tellings, model/digest/engine_version/prompt_version/config provenance. A telling's `produced_by` must be consistent with — not contradict — its parent reading's `produced_by`.
- `schema_version` — present on every new telling record; un-retrofittable. (No literal version string assigned here — a mechanical/adoption detail for the implementing change.)

**Operation identity (canonical naming)**
- `telling_idempotency_key` — **the one canonical record-level idempotency identity of a telling card.** Present and unique per store. The writer rejects an already-committed `telling_idempotency_key`. Fixed in the envelope (§4) so commit and recovery use the same key.
- The **wider operation key** is a separate, distinctly-named device: `telling_creation_operation_key` (§4) scopes the *creation operation for one telling within one reading-pass*, from which the record-level `telling_idempotency_key` is **deterministically derived**. They are not interchangeable: the operation key identifies the *operation*; the idempotency key identifies the *record*. Scopes and relationship in §4.

**Lifecycle (no mutable status in the immutable core)**
- The immutable telling core carries **no editable `storage_status` field.**
- A telling's **effective lifecycle eligibility** (quarantine, production, visibility, rejection, retention) is **derived** from its parent reading's valid append-only lifecycle state and/or represented through **separate append-only lifecycle events** that point to the `telling_id` — never by mutating the telling card. Full model in §5.

**What this contract must never include:** no `reason`-as-asserted-fact field (the "why" is shown by pointers, never stored as asserted fact — wiring map Part 7.5); no field that copies root text or another telling's content; no field that records a winner, resolution, or merge of conflicting tellings; no field that turns the telling into a profile, diagnosis, or settled fact; no mutable status field of any kind in the immutable core.

**Explicitly not decided here (open, named):** firmness scale/labels/thresholds; evidence-relationship weighting; any identity threshold for deciding two perspective references are the same person (Person-Box / A5 territory, not telling identity); theme-confirmation policy. This contract defines *identity and shape*, not *evidential or interpretive policy*.

# 3. RELATIONSHIP TO THE EXISTING BUILT READING CONTRACT — WITH CORRESPONDENCE/INTEGRITY INVARIANT

**The fact base (Master §6B line 541, lines 535–545; Decision Defaults lines 58–62):**
- The built, verified v1 reading record has 12 fields: `id · reads · meaning · confidence · role · story_layer · mode · timestamp · produced_by · schema_version · derived_from · idempotency_key`.
- `story_layer` is a **LIST** of tellings; an empty list is legal.
- Each existing embedded telling entry carries optional `whose · stance · firmness · telling · theme · when`; unknown sub-values are omitted, never `"unknown"`.
- Existing reading records are **immutable**. Existing embedded telling content must not be rewritten or deleted.

**The single relationship this package defines:** the first-class telling record is the **permanent telling-level link target**. The embedded `story_layer` entry inside the immutable reading remains exactly as written — preserved, never rewritten, never deleted. The first-class record does not replace the embedded entry; it is the addressable, linkable object that carries the embedded entry's content forward under a stable identity, plus the required provenance and structured-perspective fields the flat embedded form did not have.

Governing rules:
1. **The embedded entry is the original record of what the engine pass produced.** It stays inside its immutable parent reading as the provenance anchor. Never edited to match the first-class record. *(Both the embedded entry and the parent reading are themselves fixed in the durable envelope before commit — §4 — so the embedded form is preserved exactly from acceptance onward.)*
2. **The first-class telling record is the canonical link target — only when it is integrity-valid (rule 5) and the §5A semantic-eligibility gate is satisfied.** Every downstream component that needs to point at "a telling" points at the `telling_id`, never at "reading X, list position N."
3. **Content correspondence is one-directional and append-only.** A first-class record corresponds to exactly one embedded entry, identified by `embedded_entry_correspondence` (§2). The embedded entry inside the immutable reading is the historical truth of what the pass emitted.
4. **There are never two editable authorities.** Both the embedded entry and the first-class record are immutable once written. Neither is a mutable cache of the other. A later interpretation does not edit either; it creates a *new* telling record (new `telling_id`) beside them.
5. **Correspondence/integrity invariant.** A three-way correspondence must hold and be verifiable across:
   - the durable **`prepared_reading_telling_envelope`** entry (§4);
   - the **embedded reading entry** it was derived from;
   - the committed **first-class telling card**.
   Verification is by **stable identity** (`telling_id` + `embedded_entry_correspondence`), **deterministic correspondence** (the card's carried-forward embedded values equal the embedded entry's values, field by field per §4A), and an **integrity digest** (entry-level, plus the whole-envelope digest — §4) re-checkable against the committed card. Because the envelope fixes the exact embedded `story_layer` and the exact card payload **before** the reading commits, all three sources derive from one durable origin; **any unexplained mismatch is an integrity failure.**
6. **Divergence fails closed.** A first-class card that fails the integrity invariant **must fail closed: it must not be available as a canonical downstream link target** for semantic use. The failure is recorded (§10) without rewriting any immutable source — neither the embedded entry, the parent reading, nor any committed record is altered. N.H **never silently reconciles, normalizes, or chooses the divergent card over the original embedded output.** The embedded entry remains the historical truth; the divergent card is barred from canonical semantic use until resolved through an authorized path (not invented here). *(An integrity-failed card is nonetheless still discoverable and governable by §7Q and security mechanisms — §5A.)*
7. **The complete first-class *semantic* payload passes acceptance before the manifest exists; system fields are mechanically assembled after acceptance.** The embedded v1 `story_layer` entry does not contain every field the first-class card requires (it carries only `whose · stance · firmness · telling · theme · when`). The **semantic** card fields — the SMART perspective roles (`subject`, `perspective_owner`, optional `attribution_path`), `evidence_relationship`, `telling`, `stance`, `firmness`/`firmness_evidence_basis`, `source_event_time`, and the support claim expressed through pass-local evidence handles — **must not bypass the Reading Proposal Acceptance Check.** Therefore:
   - **The mouth/SMART proposal supplies the complete semantic telling material** for each expected telling — the genuinely interpretive perspective roles (`subject`, `perspective_owner`, optional `attribution_path`), `evidence_relationship`, `telling`, optional `stance`, optional `firmness`/`firmness_evidence_basis`, optional `source_event_time`, and the support claim expressed through **deterministic pass-local evidence handles** — **plus** the embedded story entry. It does **not** supply `root_speaker` or parent-reading `role` (both source-carried), persistent raw root IDs, system identity, idempotency/operation keys, system timestamps, schema/version, or mechanically-derived correspondence identifiers — those are not mouth claims (§2.1).
   - **The acceptance check validates every semantic claim and grounding relationship that will enter the card** — not merely the bare reading and not merely the flat embedded entry. A semantic field not grounded in the target root or supplied context fails the check.
   - **After semantic acceptance, the DUMB/system layer mechanically assembles the system fields** (`telling_id`, `reading_id`/`parent_reading_id`, reading `idempotency_key`, `telling_creation_operation_key`, `telling_idempotency_key`, `preparation_id`/`preparation_idempotency_key`, the resolved `supporting_root_ids`, `embedded_entry_correspondence`, `schema_version`, the entire DUMB `derivation_provenance`, the deterministic execution/model/config provenance assembled from the actual pass, and **only the card's `created_at`**) and performs **shape, uniqueness, provenance-consistency, and referential-integrity validation** **before** manifest durability. **It does not assign `prepared_at` or `committed_at` here** — `prepared_at` is created later by the manifest-validity event and `committed_at` later by the card-commit event (§2.0 E).
   - **No semantic field may be added, inferred, normalized, replaced, or enriched after acceptance.** Mechanical record assembly after acceptance **is** permitted and **is not** semantic enrichment — it attaches machine-state/provenance facts (DUMB), it never alters or supplements the accepted interpretation (SMART).
   - **The manifest binds both** the exact accepted semantic payload **and** the exact mechanically-assembled identity/provenance fields (§4); the embedded entry, the manifest's semantic payload, and the committed card's semantic content are **all derived from the same accepted proposal**, never from three separate generations.
   - **A proposal that fails a required acceptance condition is rejected — not partially salvaged.** The Reading Proposal Acceptance Check applies to the **proposal**; a proposal that fails a required condition is **rejected**, its specific failure reason is recorded (Master §7G: fabrication / malformed / incomplete / unsupported-certainty / mode-violation are distinct rejection reasons — Master §7G lines 1690, 2009), and the rejected material **never silently becomes the reading.** **Until B24 and B9 are designed, a substantively rejected proposal is terminal.** A2 invents **no** partial salvage, field stripping, or removal of one invalid telling while accepting the rest: **no invalid or ungrounded telling may enter the manifest or become a card**, and an **empty `story_layer` must never be manufactured by silently deleting a rejected telling.** Whether any future validator architecture supports partial proposal repair, a fresh proposal, or another bounded behavior remains **B24/B9** and is not decided here. A telling does **not** become an accepted card through manifest creation or mechanical assembly; those persist and identify an already-accepted payload, never substitute for acceptance.
   - **Two distinct checks, never merged:** the **acceptance result** proves *grounding and substantive eligibility of the semantic payload* (validated once, before the manifest); the **integrity validation** (rule 5) proves *byte/field correspondence* among manifest, embedded entry, and card (validated at/after commit), and the mechanical-assembly validation proves *shape/uniqueness/provenance-consistency/referential-integrity of the system fields*. None stands in for another. This package consumes "the acceptance check passes" as a precondition and does **not** design the validator itself (that is B24); it invents **no** new evidential threshold or grounding policy (A31 / A-policy).
8. **Deterministic pre-manifest correspondence validation (Corrections 4 + the prior pre-manifest stage — runs after semantic acceptance, before manifest validity).** The accepted proposal carries two semantic representations of each telling — the embedded v1 `story_layer` entry and the complete first-class telling semantic payload. Before the manifest may become valid (and therefore before the reading or any card commits), a **deterministic correspondence-validation stage** verifies that the **complete card payload actually contains the §4A preserved legacy values** and that they correspond exactly:
   - the **embedded telling count and first-class telling count match**, and **one embedded entry maps to exactly one first-class payload** (`embedded_entry_correspondence` resolves one-to-one, order/discriminator unambiguous);
   - **every populated legacy value is preserved exactly on the card** — `legacy_whose` (from embedded `whose`), `stance`, `firmness`, `telling`, `legacy_theme` (from embedded `theme`), `embedded_when` (from embedded `when`);
   - **omission remains omission** (no value invented where the embedded entry omitted one);
   - **no normalization or replacement** of any populated value;
   - **no first-class payload loses a populated embedded field**;
   - a **semantic field with no embedded-v1 counterpart** (the SMART perspective roles, `evidence_relationship`, etc.) may exist in the first-class payload **only because it passed the substantive acceptance check** (rule 7) — never because correspondence "filled it in".
   **If this pre-manifest correspondence check fails:** the manifest **must not become valid**; the reading **must not commit**; **no card may commit**; **no telling-specific semantic fan-out** occurs; a **distinct structural/correspondence failure reason** is recorded (§10); and **no representation is chosen as the winner**. The failure is then **classified by its actual cause**: a malformed or substantively divergent proposal is a nonretryable `failed` (terminal, unless an already-authoritative rule says otherwise) and a technical validator/storage/integrity-system failure is a potentially-retryable `failed` held pending B9 — **both governed by §4.9**; only an actual crash/interruption before a lifecycle event is an `abandoned` + RC-4 — **governed by §4.4.1A**. **A correspondence failure does not automatically enter a new RC-4 pass.** This stage **prevents a known divergence from ever being committed.** It is **separate from** the post-commit integrity re-check (rule 5), which later proves the *committed bytes still match the already-validated manifest* and so catches *write/storage* divergence. The pre-manifest check guards the accepted-proposal→manifest boundary; the post-commit check guards the manifest→durable-record boundary.

**Forward path (Engine C emits fresh):** when Engine C's pass produces a proposal covering both the reading and, for each expected telling, the complete **semantic** payload, and that **semantic proposal passes the Reading Proposal Acceptance Check** (rule 7), the DUMB layer mechanically assembles the system fields (§4.3 step 3) and then the exact accepted semantic payload, its exact embedded `story_layer`, and the exact assigned identities/provenance for every expected telling are all fixed in the durable manifest **before** the reading is appended (§4). The reading and each card are then appended idempotently from the manifest. The integrity invariant (rule 5) binds manifest, embedded entry, and card; the acceptance result (rule 7) — established once, before the manifest — establishes grounding of the semantic content.

**Materialization path (first-class records from pre-existing embedded entries):** any future materialization must be append-only (writes new cards; never edits/deletes embedded entries or parent readings), auditable (records that it was materialized and from which embedded entry), deterministic/idempotent (re-running over the same embedded entry creates no second card — §4), never a rewrite of the parent reading, and **subject to the fail-closed legacy-materialization rule of §4A** (an incomplete legacy entry never becomes a valid card by invention).

**This package performs no migration, no backfill, and no disk modification.** Materialization is target architecture only; whether/when/under what authorization it runs is a later separately authorized act.

# 4. CREATION, PERSISTENCE, AND EXACT-PAYLOAD CRASH RECOVERY (implementation-neutral)

All design, not code. No pseudocode that could be mistaken for patch-ready code.

## 4.1 The durable preparation manifest (prior rounds: preparation before reading, acceptance-before-manifest; this round: semantic/mechanical two-part contents + prepared_at on the validity event)

**Why it exists, and why preparation precedes the reading commit.** v1.1 committed the parent reading first and wrote its recovery payload second, leaving a crash boundary where the reading was durable but no recovery payload was — at which point the complete structured telling payload was lost and recovery could only proceed by regenerating or reinterpreting the telling, which v1.1 itself prohibited. v1.1 then permitted "a re-run of the producing pass," contradicting its own no-rerun rule. The prior rounds removed that contradiction by moving preparation **before** the reading commit and requiring the semantic payload to pass the acceptance check before the manifest. This round adds that the manifest's contents are explicitly **two parts** — the accepted *semantic* payload and the *mechanically-assembled* identity/provenance fields — assembled in that order (§4.3), and that `prepared_at` lives on the validity event, not the payload (§2, Issue 3). This sits cleanly in the settled pipeline: the post-root path's order is `append_root()` → enqueue → mouth proposal → **Reading Proposal Acceptance Check passes** → `append_reading()` to quarantine (map lines 356–369; Master §7G-A lines 1677, 2009). Semantic acceptance, then mechanical assembly, then the durable manifest, all sit between "Acceptance Check passes" and `append_reading()`, as crash-safe acceptance persistence.

**What it is.** An implementation-neutral, durable, **immutable, operation-scoped `prepared_reading_telling_envelope`** for one reading operation. Created **after the Reading Proposal Acceptance Check passes and after the DUMB/system layer has mechanically assembled the system fields, but before the parent reading is durably appended.** It preserves the **exact accepted semantic output and the exact mechanically-assembled identity/provenance fields** of that reading operation so that nothing accepted is ever lost and nothing ever needs regenerating. It contains, at minimum — in two clearly separated parts:

*Accepted semantic part (SMART; grounded by the acceptance check — §3 rule 7):*
- the **exact immutable embedded `story_layer`** as accepted;
- the **exact accepted semantic content** of the parent-reading record, preserved exactly under the existing authoritative twelve-field reading contract (its interpretive content such as `meaning`; the source-carried `role`; and the two-slot `confidence`, which **remains reading-level metadata** — A2 does not decide or reclassify how that reading field is produced, never copies it onto the telling card, and keeps telling firmness separate from it — §2.0 R);
- the **exact complete semantic payload for every expected telling** (the SMART perspective roles `subject`/`perspective_owner`/optional `attribution_path`, `evidence_relationship`, `telling`, optional `stance`, optional `firmness`/`firmness_evidence_basis`, optional `source_event_time`, and the accepted support claim expressed through pass-local evidence handles).

*Mechanically-assembled part (DUMB; attached after acceptance, validated for shape/uniqueness/provenance-consistency/referential-integrity):*
- deterministic `preparation_id` and `preparation_idempotency_key`;
- the **parent reading operation identity** (the pass's operation identity / its `reading_idempotency_key` derivation, tying manifest to producing pass);
- the **exact assigned `reading_id`** and the reading's **`idempotency_key`**;
- the **expected telling count** and the **ordered stable per-telling discriminators**;
- **every assigned permanent `telling_id`**, **every `telling_creation_operation_key`**, **every `telling_idempotency_key`**;
- **`embedded_entry_correspondence`** (mechanically-derived) for every telling;
- **`schema_version`** and the deterministic **execution/model/config provenance** assembled from the actual pass;
- **`created_at`** for every telling (logical creation timestamp, assigned during mechanical preparation before the digest is finalized — §2);
- an **integrity digest for the whole manifest and for every contained reading/telling payload**.

*Validity boundary:* the **`prepared_at`** timestamp is **not** stored inside the payload above — it is recorded on the separate **append-only manifest-validity/completion event** (§4.2) that proves this manifest became durable. The complete manifest (both parts + digests) is what the validity event certifies.

**The envelope must be durably valid before the parent reading may commit.** This is the crash-safety guarantee: no committed reading can normally exist without its exact durable recovery envelope already present, so that once a reading is durable its telling cards are recoverable from the manifest (§4.4 case C makes the one anomalous exception — a committed reading whose manifest is absent — an integrity violation, never a rerun trigger).

**Recovery of a committed reading's cards uses the manifest, never the model.** When the parent reading is **already durable**, recovery reads the exact manifest associated with that reading and commits the **missing telling cards, commit events, integrity checks, and the completion checkpoint** idempotently from the preserved exact payloads under their **assigned** identities. **Recovery must never rewrite the reading or rerun the model to reconstruct cards belonging to that committed reading.** When the parent reading is **not yet durable** and the pass was crash-interrupted (no terminal lifecycle event, no reading at the job-level `reading_idempotency_key`), recovery does **not** use the manifest to commit a reading — it follows **Master V10's RC-4** (append one `abandoned` event, create a fresh linked pass that re-runs from Step 2), and the abandoned manifest stays preserved under §0B/§7Q but is never used to continue the abandoned pass (§4.4).

**The manifest is protected recovery material, not an ordinary semantic-memory or evidence source** (prior round: §7Q wording; full rules in §4.2 and §9). A prepared payload — and a reserved-but-uncommitted identity — is preparation, not a committed first-class telling. It **must never** be used as semantic evidence, context, a Story-Layer object, a Person-Box source, a clash input, a theme input, a Living-State-Web source, a Computed-View source, or an LMAC retrieval source. Only a committed, integrity-valid, gate-satisfied first-class card is a usable semantic link target.

## 4.2 Manifest identity, atomicity, and duplicate prevention (prior round: manifest-as-protected-object)

The envelope is a real protected stored object and has complete mechanical protection:

- **One deterministic envelope operation identity:** `preparation_id`, derived deterministically from the producing pass's operation identity (so one accepted reading operation maps to exactly one preparation).
- **One canonical envelope idempotency key:** `preparation_idempotency_key`, deterministically derived from `preparation_id`; the writer rejects an already-committed key.
- **Atomic validity:** a partially written envelope is **never** treated as valid. The envelope becomes valid only when it is durably complete and passes its whole-envelope integrity digest. If the physical representation cannot be one atomic record, an **append-only completion/validity record** marks the envelope valid only after all contained payloads and digests are durably present; until that record exists, the envelope is treated as not-yet-valid (incomplete), and the reading must not commit from it.
- **Expected-count and ordered-discriminator inventory:** the envelope records the expected telling count and the ordered stable per-telling discriminators, so completeness is checkable, not inferred.
- **Whole-envelope integrity digest** plus **entry-level integrity digests** (one per contained reading payload and per contained telling payload), enabling both whole-object and per-entry integrity verification.
- **Duplicate prevention on retried preparation:** a retry under the same `preparation_id` / `preparation_idempotency_key` either **finds the identical valid envelope and performs an idempotent no-op**, or **detects a content mismatch and fails closed**. It **never** creates a second preparation envelope for the same accepted reading operation.
- **The manifest-validity event contract (Correction 3A).** The append-only manifest-validity event that certifies the complete manifest carries: **one stable operation identity derived from `preparation_id`**, **one idempotency key**, and **duplicate-rejection / idempotent-no-op behavior** (a repeated append finds the existing event and is a no-op). **Within a still-active pass** (the worker is alive, has written all manifest bytes and digests, and has not yet appended the validity event), recovery may append the missing validity event idempotently — no rerun, no replacement ids. **This in-pass recovery must never be used to bypass Master V10's RC-4 after a crash/restart:** once the pass has crashed (the worker exited), an instruction with no terminal lifecycle event and no reading at the job-level `reading_idempotency_key` is handled by RC-4 (abandon + new pass), regardless of how complete the manifest bytes are (§4.4.1).
- **Recovery after a crash during manifest creation follows Master V10's RC-4 (Correction 1).** Whether the crash left nothing, a partial manifest, or complete manifest bytes with the validity event absent, the rule is the same when an instruction exists, no terminal lifecycle event exists, and **no reading exists at the job-level `reading_idempotency_key`**: append **exactly one `abandoned`** lifecycle event with the crash/interruption `abandonment_reason`, then create a fresh linked pass per §4.4.1B that re-runs from Step 2. The job-level `reading_idempotency_key` prevents duplicate readings. The abandoned manifest/partial artifact is **preserved under §0B/§7Q** (never destroyed — §9) but is **never used to continue the abandoned pass** and is never treated as semantic evidence. A detected malformed/corrupt manifest that is **not** a crash-interrupted pass is a `failed` event handled per §4.9 (technical → held for B9; substantive → terminal), never RC-4. **No identifier already durably recorded in the abandoned/failed partial artifact may later identify a different reading, telling, preparation, or event** (spent identities stay spent — §2).
- **The durable no-loss boundary applies to a committed reading's cards.** The manifest guarantees that once a reading is durable its telling cards are recoverable from the exact preserved payloads. Before the reading commits, an in-memory or partially-durable proposal is not a protected no-loss object; a crash there is an RC-4 abandon-and-replace, not a payload to be recreated under an existing identity.
- **Fail-closed behavior** for partial, malformed, duplicate-conflicting, or integrity-failed envelopes: the envelope is not treated as valid; the parent reading does not commit; no telling card is committed; the anomaly is recorded (§10) without leaking payload content (§9); no recovery payload is invented.

## 4.3 Correct commit sequence (semantic acceptance → mechanical assembly → durable manifest → commit)

1. **Produce the semantic proposal** for the reading **and its tellings** — the reading's semantic content plus, for each expected telling, the complete **semantic** payload (§2 semantic fields) and the embedded story entry. The mouth supplies **no** system identity/idempotency/timestamp/schema fields.
2. **Validate the semantic proposal through the applicable acceptance checks** (the independent Reading Proposal Acceptance Check — Master §7G lines 1690, 2009 — applied to every semantic field that will enter a card, not just the bare reading; §3 rule 7). A proposal whose required semantic fields are not grounded is **rejected** with its specific reason recorded; it does not proceed, and (until B24/B9) is terminal. **No invalid telling is stripped so the rest can proceed, and no empty `story_layer` is manufactured by deleting a rejected telling.** An accepted **zero-telling** reading is valid only when the accepted proposal itself legitimately contains an empty `story_layer`.
3. **Mechanically assemble the system fields (DUMB layer), after acceptance:** assign `reading_id` and the reading `idempotency_key`; assign each `telling_id`, `telling_creation_operation_key`, `telling_idempotency_key`; derive each `embedded_entry_correspondence`; resolve each telling's exact `supporting_root_ids` from the accepted evidence handles and validate existence/pass-availability/`reads`-consistency (§2.1); copy the accepted legacy values (`legacy_whose`/`legacy_theme`/`embedded_when`) exactly without reinterpretation; assign `schema_version`, `produced_by` (the reproducibility-grade producer/model/config provenance), the DUMB `derivation_provenance` (lifecycle type, pass id, recovery lineage, materialized source, execution-path identity — referencing `produced_by`, not duplicating model/config), and **only the card's `created_at`** among timestamps. **`prepared_at` and `committed_at` are not assigned here** — they are created later by their events (§2.0 E). **Validate shape, uniqueness, provenance-consistency, and referential integrity** of these system fields. **No semantic field is added, inferred, normalized, replaced, or enriched in this step.**
4. **Run the deterministic pre-manifest correspondence validation (§3 rule 8):** verify embedded-vs-first-class telling count, one-to-one order/discriminator correspondence, exact §4A legacy-value preservation on the card (`legacy_whose`/`stance`/`firmness`/`telling`/`legacy_theme`/`embedded_when`), omitted-stays-omitted, no normalization/replacement, and that no first-class payload loses a populated embedded field. **On failure the manifest does not become valid, nothing commits, no fan-out occurs, a distinct structural/correspondence failure reason is recorded (§10), no representation is chosen as winner, and the failure is classified by cause** (divergent → nonretryable `failed`; technical → retryable `failed` held for B9 — both per §4.9; actual crash → `abandoned` + RC-4 per §4.4.1A) — **never auto-RC-4.** For a **zero-telling** reading this stage confirms expected count zero and passes trivially.
5. **Make the manifest durable** (`prepared_reading_telling_envelope`) under its deterministic `preparation_id`, binding **both** the exact accepted semantic payload **and** the exact mechanically-assembled fields, plus the whole-manifest + entry-level integrity digests; then append the **manifest-validity/completion event** carrying `prepared_at` (§4.2). **A parent reading cannot become durably committed unless this manifest is already valid** — the ordering invariant that eliminates the committed-reading/missing-manifest crash window. **This applies to every accepted reading, including a zero-telling reading** (its manifest records expected count zero and supports the zero-telling completion sentinel; empty `story_layer` never bypasses the preparation protocol).
6. **Commit the immutable parent reading** idempotently from the manifest's exact payload (using the assigned `reading_id`/`idempotency_key`). A crash before this point, with no reading at the job-level `reading_idempotency_key`, is an RC-4 abandon-and-replace (§4.4.1); the manifest is not used to commit a reading after a crash.
7. **Commit each first-class telling card** idempotently from the manifest's exact payload, each under its assigned `telling_id`, `telling_creation_operation_key`, and `telling_idempotency_key`; append each card's commit event (carrying `committed_at`) under the card-commit event contract (§4.4.1). A card counts as committed only when the card **and** its required commit event are both durable and valid.
8. **Re-verify correspondence and integrity** of each committed card against its manifest entry and the embedded entry (§3 rule 5 — the post-commit re-check, separate from the step-4 pre-manifest check).
9. **Append the telling-completion checkpoint** (or the zero-telling sentinel) under the completion-checkpoint contract (§4.4.1), only after every expected card and its commit event exist and all integrity checks pass; telling-specific semantic fan-out becomes eligible only then (§5A).

**Wait/independence rules.** The reading commit **may wait** for the durable, valid manifest — that manifest is part of crash-safe acceptance persistence, and by invariant must already be valid before the reading commits. The reading commit **must not wait** for the later card writes (step 7) or telling-specific fan-out (§5A). **Root ingestion remains complete independently** (`append_root()` is durable upstream and never blocks on the mouth — Master §7G-A); the later card creation remains **asynchronous and recoverable** from the manifest of a committed reading.

## 4.4 Crash recovery follows Master V10's RC-state machine (Correction 1)

Recovery uses **Master V10's authoritative RC-1–RC-8 recovery rules exactly** — this package does not introduce a separate continuation/cancellation state machine. Before any RC case, the worker first checks for a **handled technical failure** (§4.9): if the latest pass-lifecycle event is `failed` with `is_technical_and_potentially_retryable = true`, this is **not** a crash and recovery does not proceed — it is held for B9. The immutable parent reading is **never rewritten**, and **no statement permits a new producing/model pass to reconstruct missing payloads or "re-derive the same identities" under an existing identity.**

The A2-relevant cases:

- **No instruction exists (RC-3 territory).** Create the instruction as Master V10 requires (run `thread_membership_v1` fresh, validation check, append the instruction entry), then proceed.
- **Instruction exists, no terminal pass-lifecycle event (RC-4 check).** Compute and check the deterministic job-level `reading_idempotency_key`:
  - **If the reading already exists** → it is **not** abandoned; proceed through **RC-5** (recover `reading_id`, append the `completed` lifecycle/`reading_written` records if absent) and the **A2 post-reading card-recovery extension** (case C below).
  - **If the reading does not exist** → append **exactly one** `abandoned` pass-lifecycle event with the authoritative crash/interruption reason, then create a fresh linked pass (§4.4.1B) and run the full sequence from Step 2. This applies after **any** crash before the reading is durable, including: semantic acceptance only in memory; mechanical assembly begun; a partial manifest; complete manifest bytes with the validity event absent; and **a valid manifest whose reading never committed.** The old manifest/partial artifact stays preserved under §0B/§7Q but is **abandoned with its pass** and never used to continue it.
- **Reading exists — A2 post-reading card recovery (case C).** Using the **exact manifest associated with that committed reading**, recover idempotently: the **missing telling cards**, the **missing card-commit events**, the **integrity checks**, and the **completion checkpoint** — each under its assigned identity. Cards/events already present are skipped. **Never rewrite the reading or rerun the model to reconstruct cards belonging to that committed reading.**
- **Handled technical `failed`** → held for B9 (§4.9): no RC-4 replacement, no automatic rerun, no automatic re-attempt; release the claim/lock as Master V10 governs.
- **Substantive/nonretryable `failed`** → terminal (§4.9): no replacement, no automatic rerun; preserve the exact `failure_stage` and reason.

**Crash matrix (every state explicit).** Every crash **before the reading is durable** resolves the same way: if **no instruction exists**, use **RC-3** (create the instruction); if an **instruction exists, no terminal lifecycle event exists, and no reading exists at the deterministic job-level `reading_idempotency_key`**, use **RC-4** (append one `abandoned` lifecycle event; create a fresh linked pass; preserve the same `job_id` and job-level reading idempotency key; use new pass/preparation/candidate record identities; preserve `recovery_of_pass_id`). This holds whether the crash occurred **before the mouth proposal, during proposal creation, before acceptance, or after acceptance but before reading commit.** **Recovery must never rerun the model to recreate the same accepted output under the same identities;** the authoritative RC-4 fresh linked replacement pass **is allowed and required** precisely because it is a *new* pass with *new* candidate identities (not a re-derivation under an existing identity). The named states:
- *No instruction yet (before/at proposal)* — RC-3: create the instruction, then proceed.
- *Before semantic acceptance, instruction exists, no reading* — RC-4 abandon + fresh linked pass (new candidate identities); no manifest/reading/card exists yet.
- *After acceptance, before/in mechanical assembly, no reading* — RC-4 abandon + new pass; assigned-but-never-durable identities are never used; durably-written identities are spent.
- *Partial manifest, no reading* — RC-4 abandon + new pass.
- *Manifest bytes complete, validity event absent, pass still actively owned* — append the validity event idempotently in-pass (§4.2 manifest-validity contract); **after a crash** with no reading → RC-4 abandon + new pass (no continue-from-manifest).
- *Valid manifest, reading never committed (after crash)* — RC-4 abandon + new pass; the manifest is abandoned, not used to commit a reading.
- *Reading committed, no cards* — case C: commit the exact cards + commit events from the manifest under assigned identities.
- *Card exists, its commit event absent* — case C: append the missing commit event idempotently (§4.4.1 card-commit contract); the card is not "committed" until its commit event exists.
- *Commit event exists, card absent* — **fails closed as an invariant violation** (§4.4.1): recorded (§10) without content leak (§9); never papered over by minting a card.
- *All cards and commit events exist, completion checkpoint absent* — case C: verify the complete set against the expected-count/discriminator inventory and integrity digests, then append **only** the checkpoint.
- *Completion checkpoint already exists on retry* — idempotent no-op (no duplicate checkpoint).
- *Zero-telling sentinel already exists on retry* — idempotent no-op (no duplicate sentinel).
- *Reading exists but its required manifest is absent* — an **invariant/integrity violation**, not a rerun trigger: fail closed, block telling-level semantic fan-out (§5A), record the failure (§10) without leaking content (§9); the reading stands (immutable) but its tellings cannot be reconstructed (exact accepted payloads are the only permitted source). Any future reinterpretation is a genuinely new reread producing a new reading and new telling identities (§8). Resolution is an authorized-recovery/audit matter (§9). *(Because §4.3 makes a valid durable manifest a precondition of reading commit, this should not arise in normal operation; it is a containment/integrity case.)*

**Failure-classification cross-reference (Correction 4).** Crash/interruption recovery is governed by the RC cases above — **§4.4.1A** governs the RC-4 `abandoned` abandonment and its replacement lineage. **Non-crash `failed` outcomes are governed by §4.9** — a handled technical `failed` is held for B9, and a substantive `failed` is terminal. These are distinct: an `abandoned` event is never used where a `failed` event belongs, and vice versa.

**General fail-closed rule.** On any system failure: no partial, malformed, or integrity-failed card is left as if valid or canonical; no completion checkpoint is appended while any expected card or its commit event is absent or invalid; a committed reading's missing cards are recoverable from its manifest under assigned identities; the system never reports completion on failure; a half-written or divergent card is never exposed as a usable semantic link target.

## 4.4.1 Abandonment, RC-4 lineage, and event/checkpoint contracts

**A. The `abandoned` pass-lifecycle event is the single record for an abandonment (Correction 4).** When the RC-4 check determines a crash-interrupted pass with no reading at the job-level `reading_idempotency_key`, the worker appends **exactly one** `abandoned` pass-lifecycle event with the authoritative crash/interruption `abandonment_reason`. There is **no separate "abandonment transition" log** — the `abandoned` event *is* the abandonment operation's one record. Creating the replacement RC-4 pass is a **later, genuinely separate operation with its own log** (§10). A handled technical or substantive `failed` pass is closed by a single `failed` event (§4.9), not by an `abandoned` event, and creates no replacement.

**B. Exact recovery lineage for the RC-4 replacement pass (crash-interrupted `abandoned` only).** The replacement pass:
- **preserves the same queue `job_id`**;
- **preserves the authoritative deterministic job-level `reading_idempotency_key`** across replacement passes (the explicit RC-4 duplicate-prevention exception — Master: `reading_idempotency_key = stable_hash(enqueue_key + "reading_v1")`, the same job-level key across attempts);
- allocates a **new `pass_id`**, a **new `preparation_id`**, a **new candidate `reading_id`**, **new candidate `telling_id`s**, **new pass/preparation-scoped operation identities**, and **new candidate `created_at` values**;
- **records `recovery_of_pass_id = [abandoned pass_id]`** linking the new pass to the closed prior pass.

`recovery_of_pass_id` is used **only** for the new pass replacing an `abandoned` crash-interrupted pass; it never implies that a `failed` pass automatically receives a replacement. **Any identity durably recorded in the abandoned partial artifact is permanently spent** and may not later identify a different reading, telling, preparation, or event. The **job-level `reading_idempotency_key` is the one explicit exception** — it deliberately stays the same across replacement passes to prevent duplicate readings, and **must not be confused with reuse of the abandoned record identities** (which never recur).

**C. The telling-card commit event contract (Correction 3B).** Each append-only commit event carrying `committed_at` has:
- a **stable event identity**;
- an **idempotency key deterministically tied to the card's `telling_idempotency_key`**;
- either an **explicit transaction boundary with the card append**, or a **separately recoverable event append** (so the card and its event can be reconciled after a crash);
- **no duplicate commit events after retry** (a repeated append is an idempotent no-op).

Recovery states for a card and its commit event: *card absent / event absent* → commit both from the manifest under the assigned identity; *card present / event absent* → append the missing commit event idempotently; *card present / event present* → no-op; *event present / card absent* → **fails closed as an invariant violation** (recorded without content leak; never resolved by minting a card). **"Card committed" means the card and its required commit event are both durable and valid** — a card without its commit event is not yet a committed, usable telling.

**D. The telling-completion checkpoint / zero-telling sentinel contract (Correction 3C).** There is **one deterministic completion-operation identity for the reading's telling set**, derived from stable reading/preparation identity plus a completion-operation-version tag. The checkpoint (or zero-telling sentinel) carries: the **completion-operation identity**; **one idempotency key**; the **parent reading ID**; the **expected telling count**; the **verified expected discriminator/card inventory or its integrity digest**; a **result type** (`complete-card-set` or `legitimate-zero-telling`); a **timestamp**; and **schema/version provenance**. **Retries append no duplicate checkpoint or sentinel.** The checkpoint may be appended **only after every expected card exists, every required card-commit event exists, and all correspondence/integrity checks pass**; the zero-telling sentinel is appended for an accepted zero-telling reading whose manifest records expected count zero. The completion checkpoint/sentinel is the **single log** for telling-set completion (§10).

## 4.5 Zero, one, or several tellings

- **Zero tellings** (empty `story_layer`, legal): valid **only when the accepted proposal itself legitimately contains an empty `story_layer`** — an empty set is **never manufactured by silently deleting a rejected telling** (§3). The envelope records expected count zero; no cards are committed; the checkpoint records a **zero-telling sentinel** — an explicit positive completion fact, not silence — distinguishing "legitimately no tellings" from "creation never ran." The zero-telling sentinel satisfies the §5A semantic-eligibility gate.
- **One telling:** the envelope carries one telling payload; one card committed; one completion checkpoint.
- **Several tellings:** the envelope carries one payload per telling under their assigned identities; each card committed under its own identity; the single completion checkpoint appended only after **all** are durably committed and integrity-valid. Conflicting tellings within the same reading remain separate cards — never merged.

## 4.6 Operation identity / idempotency (canonical naming)

- `telling_creation_operation_key` — scopes **one telling-creation operation within one reading-pass.** Derived deterministically from stable inputs that uniquely identify *that telling of that reading*: at minimum the `parent_reading_id` (the predetermined `reading_id`), the stable per-telling discriminator (`embedded_entry_correspondence`), and a telling-operation version tag. Fixed in the envelope. Mirrors the existing operation-keyed pattern in the post-root path (e.g. `reading_idempotency_key = hash(enqueue_key + "reading_v1")`, and operation-keyed clash/Computed-View sentinels).
- `telling_idempotency_key` — the **record-level** idempotency identity stored on the card, **deterministically derived from** `telling_creation_operation_key` (e.g. a further keyed derivation tagged `telling_v1`). Unique per store; the writer rejects an already-committed key. Fixed in the envelope.
- **Relationship and scope:** the operation key identifies the *operation* (what recovery re-derives — though here it is already fixed in the envelope — to commit the right card); the idempotency key identifies the *record* (what the writer's duplicate check rejects). One operation key yields exactly one idempotency key. `telling_id` is a third, separate thing: the **permanent identity**, allocated once at preparation and preserved across all lifecycle states and locations.

## 4.7 Duplicate prevention

Before committing a card, the writer checks whether a card with that `telling_idempotency_key` already exists. If it does, the existing card stands and no second card is written — idempotent. The same rule governs materialization. Because identities are assigned in the manifest, recovery and retry always reference the same keys and never produce duplicates.

## 4.8 Partial-completion detection

A reading that is committed but whose telling-completion checkpoint is absent is a partial completion. Detection relies on the per-reading checkpoint and the envelope's expected-count/discriminator inventory — not on counting cards blindly.

## 4.9 Crash/interruption recovery vs handled technical failure vs substantive failure

These three are distinct dispositions, never conflated:

- **Crash / interruption recovery** follows Master V10's **RC-1–RC-8** rules and idempotent durable-state recovery (§4.4). A crash before the reading is durable, with an instruction and no terminal lifecycle event and no reading at the job-level `reading_idempotency_key`, is an **`abandoned` + RC-4** abandon-and-replace; a crash after the reading is durable recovers the committed reading's missing cards/commit events/checkpoint from the manifest (case C). There is **no blanket rule that "transient technical failures are automatically re-attempted."**
- **Handled technical failure** — a pass with `event_type = "failed"` and `is_technical_and_potentially_retryable = true` (e.g. I/O error, lock contention, a validator/storage/integrity-system failure recorded as a handled `failed`) is **held pending the future bounded B9 retry policy**: **no RC-4 replacement, no automatic rerun, no automatic re-attempt**; the queue job stays in its authoritative nonterminal state and the claim/lock is released as Master V10 governs. Bounded retry **values** (attempts, timing, backoff, trigger) are **B9 and not chosen here**.
- **Substantive / nonretryable failure** — a pass with `event_type = "failed"` and `is_technical_and_potentially_retryable = false` (e.g. a payload that cannot satisfy the required-field contract, an integrity-invariant failure, or a substantively divergent proposal) is **terminal**: no replacement, no automatic rerun; the specific `failure_stage` and reason are preserved (§10); the engine never silently substitutes a degraded or regenerated telling, and no failed proposal silently becomes an accepted reading or telling.

**Card-stage failure is keyed to the card / telling-set operation, not to the parent pass lifecycle (Correction 5).** The card stage runs **after** the reading commits, and by then the **parent reading pass may already carry a `completed` lifecycle event** (the reading is durable). Therefore card-stage failure must **not** append a second, conflicting `failed` pass-lifecycle event against that already-completed pass. Instead:
- **Card creation and telling-set completion are distinct operations**, keyed by the existing `telling_creation_operation_key` (per card) and the telling-set completion-operation identity (per reading's telling set) — **not** by the parent pass-lifecycle event.
- **Card-stage crash interruption** is detected and recovered through the **card / commit-event / completion-checkpoint state matrix** under those same identities (§4.4 case C, §4.4.1 contracts) — idempotently, never by rerunning the model for a committed reading's cards.
- **A handled technical card-stage failure** is recorded **against the card / telling-set operation** and **held pending B9** — no automatic retry count, timing, backoff, or trigger is invented.
- **A substantive card contract or integrity failure** is **terminal for that card / telling-set completion**: the **completion checkpoint is not appended**, **telling-level semantic fan-out remains blocked** (§5A), and the failure is recorded as a distinct reason (§10).
- **The immutable parent reading and its already-`completed` pass lifecycle remain unchanged.** All failed or incomplete card-stage artifacts remain **discoverable under §7Q** (governance discovery is never blocked by the §5A semantic gate). This package defines only the implementation-neutral operation-state contract required for crash safety; it does **not** choose B9's retry policy.

## 4.10 Orphan and duplicate prevention (restated)

- **No orphan tellings:** identities and payloads are fixed in a valid durable envelope before the reading commits, and the reading commits before any card; every card carries a required `parent_reading_id` (the predetermined `reading_id`). A card can never reference a non-existent reading.
- **No duplicate cards after crash/retry:** assigned `telling_idempotency_key`s plus the pre-write duplicate check guarantee any re-run finds the existing card and writes no second one; recovery commits from the envelope under the same identities.

# 4A. EMBEDDED-FIELD NO-LOSS MAPPING AND LEGACY-MATERIALIZATION RULE

*(Preserved from v1.1 unchanged in substance.)*

## 4A.1 Field-by-field mapping contract

Every existing embedded field (`whose · stance · firmness · telling · theme · when`) maps to the first-class architecture as follows. **Every original populated value is preserved exactly; no populated value is silently reinterpreted; omitted values are preserved as honestly absent (never invented, never `"unknown"` where omission is required).**

- **`whose` →** carried forward as a **preserved legacy value** (`legacy_whose`, retained exactly as emitted). It is **not** silently mapped into `root_speaker`, `subject`, or `perspective_owner` unless the source explicitly supports that mapping. Where the source does not disambiguate which structured role the flat `whose` denotes, the structured roles that cannot be grounded are **omitted** (honest absence), and `legacy_whose` preserves the original value for inspection. (The Master itself states the flat `whose` field may remain for v1 compatibility while future schema uses the structured fields — §7K line 2198.)
- **`stance` →** preserved as its own `stance` value **explicitly**, exactly as emitted. **Never** assumed derivable from, or already contained in, the general `telling` text. Omitted when the embedded entry had no `stance`.
- **`firmness` →** carried forward as `firmness` exactly when present. If the legacy embedded `firmness` has **no recorded evidence basis**, see §4A.2 — a basis is **never invented** to satisfy the new contract.
- **`telling` →** carried forward as `telling`, unaltered.
- **`theme` →** carried forward as a **preserved legacy theme value with its provenance** (`legacy_theme`), retained exactly. **Not** automatically treated as a confirmed theme. Under the hybrid theme system (Master §7K lines 2206–2216) only Ness confirms themes; a legacy `theme` is preserved as-emitted and remains unconfirmed unless and until Ness confirms it through the theme process (not invented here). Never silently promoted to a confirmed theme link.
- **`when` →** carried forward as `embedded_when` exactly (§2), kept **distinct** from `created_at` (logical telling-record creation timestamp, in the immutable card payload), `prepared_at` (manifest-validity event timestamp), `committed_at` (card-commit event timestamp), and `source_event_time` (source/event time when supported). Omitted when the embedded entry had no `when`.

**Time concepts kept distinct (required), with the Issue 1D definitions:** `created_at` (logical telling-record creation timestamp, fixed during mechanical assembly before digest finalization, **in the immutable card payload**) ≠ `prepared_at` (timestamp of the later append-only **manifest-validity event** proving durability) ≠ `committed_at` (timestamp of the later append-only **card-commit event**) ≠ `embedded_when` (original embedded `when`) ≠ `source_event_time` (source/event time when available). **`created_at` is not equal to `prepared_at`; neither is a single "logical preparation time"; `prepared_at` and `committed_at` are event-only and never immutable card fields; and no durability time is recorded before durability occurs.** None is collapsed into another; each is omitted rather than guessed when unavailable.

**Parent-reading confidence and other uncertainty remain inspectable through `parent_reading_id`, never copied into a competing authority.** The reading's two-slot `confidence` object (`{interpretation_confidence, source_reliability}` — Master §6B line 539) and any other reading-level uncertainty are **not** copied onto the telling card. They remain on the immutable parent reading (and in the envelope's exact reading payload) and are reached by following `parent_reading_id`. One authority for reading-level confidence (the reading), one for telling-level firmness (the card), no second competing copy. Firmness on the card stays separate from the reading's model-confidence object (Master §7K line 2202).

## 4A.2 Fail-closed legacy-materialization rule

Existing embedded entries may lack fields the first-class contract makes mandatory — structured perspective roles, `evidence_relationship`, `firmness_evidence_basis`. For materialization of such entries:

- **Missing mandatory information is never invented.** No default, inferred role, fabricated evidence relationship, manufactured firmness basis, or `"unknown"` placeholder where the contract requires real content.
- **An incomplete legacy entry must not become a valid first-class telling card merely to satisfy migration.** Validity requires the mandatory fields genuinely present; an entry that cannot supply them is **blocked**, not force-fitted.
- **The blocked materialization operation is recorded with its exact missing requirements** (§10): which embedded entry, which mandatory fields are absent, why it cannot become a valid card — recorded with structural identifiers and result state, never by leaking the private payload (§9).
- **The original embedded entry is preserved unchanged**, inside its immutable parent reading exactly as emitted.
- **It remains unavailable as a valid first-class telling-level link target** until a separately valid future process — a reread that produces a new complete reading and new complete telling, a supported human annotation, or another authorized process — creates a **new complete telling** (new `telling_id`). The blocked legacy entry is never retroactively completed in place.
- **No new evidential or annotation policy is chosen here.** What counts as a valid future reread, a supported annotation, or an authorized completion process is open (A-policy / B-mechanics as applicable); this rule fixes only that incompleteness fails closed and is never resolved by invention.

*(A blocked legacy entry, like any incomplete/invalid artifact, remains discoverable and governable by §7Q and security mechanisms — §5A — even though it is barred from semantic use.)*

# 5. QUARANTINE AND PRODUCTION SEPARATION — DERIVED/EVENT LIFECYCLE

*(Preserved from v1.1 unchanged in substance.)*

**The boundary being preserved (Master/map: Engine C writes to quarantine, never directly to production; production readings store absent by design — Decision Defaults line 81, Master §6B "PRODUCTION READINGS AUTHORIZATION" lines 454–461):**

- Engine C writes readings to **quarantine**, never directly to production. Unchanged.
- **The immutable telling core is never edited when the parent reading changes state.** A telling card, once written, is immutable; lifecycle change never mutates it.
- **Effective lifecycle eligibility is derived or event-based, never a mutable field:**
  - **Derived:** a telling's effective quarantine / production / visibility / rejection / retention eligibility is **derived from its parent reading's valid append-only lifecycle state.** The reading's state is the source of truth; the telling inherits it by reference through `parent_reading_id`.
  - **And/or event-based:** where a telling-specific lifecycle fact must be represented, it is recorded as a **separate append-only lifecycle event** that points to the `telling_id` (never edited into the card). Append-only; never rewrites history.
- **A telling can never become more eligible or more promoted than its parent reading.** Structural, because eligibility derives from the parent reading; there is no independent telling promotion state that could outrun the reading.
- **The same permanent `telling_id` is unchanged across all later lifecycle states.** Promotion, rejection, retention, or visibility change never mints a new id and never retires it.
- **`telling_id` cannot collide merely because quarantine and production are separate locations.** Identity is the `telling_id` value, not the storage location. The same telling in quarantine and (if ever promoted) in production is **one identity**. (How a promotion is physically represented is part of the still-open production-promotion procedure and is **not chosen here**; the invariant fixed here is one-identity-across-locations.)
- **No production store is created here**, and **no new promotion policy is invented.** The existing production-readings authorization (Ness's deliberate `.nh_readings_production_authorized` marker plus Cursor's approved dry-run) is unchanged and untriggered by this package. Any promotion-policy detail not already settled remains **explicitly open.**

**Open here, named:** the exact telling-promotion procedure (how a telling's effective eligibility is updated/represented when its parent reading is promoted, and the operation identity of that event) follows whatever production-promotion design is later authorized. This package fixes only the invariants — immutable core, derived/event eligibility, never-more-promoted-than-parent, one identity across locations.

# 5A. SEMANTIC-ELIGIBILITY GATE vs SAFETY DISCOVERY (prior round: semantic-vs-safety distinction — preserved, not regressed)

The gate governs **semantic use**, not **safety/governance discovery**. v1.1 incorrectly listed §7Q derivative traversal among the consumers blocked by incompleteness — which would have let a partial card, integrity-failed card, or preparation envelope escape privacy/deletion/influence-removal discovery. That is corrected here: safety and governance mechanisms may always discover and govern an artifact (when properly authorized) without treating it as semantic evidence or a canonical telling.

## 5A.1 Blocked until the telling set is complete and valid (SEMANTIC use only)

Until **either** the **zero-telling sentinel** (§4.5) **or** the **complete valid set of expected first-class telling cards plus the telling-completion checkpoint** (§4.3–§4.4) exists for a reading, the following **semantic** uses may not proceed for that reading's tellings:

- Story-Layer semantic organization;
- Person-Box semantic links;
- telling-level clash analysis;
- theme links;
- Living State Web evidence use;
- Computed View use;
- ordinary retrieval;
- ordinary semantic fan-out;
- B2/B3/B4/B5 consumer work.

These must **never substitute**: an **embedded telling** for its missing first-class card; a **partial telling-card set**; or an **integrity-failed telling card** (§3 rule 6, §4.4).

**Card-stage failure keeps the gate closed without disturbing the reading (Correction 5).** When a card-stage operation fails — a handled technical failure held for B9, or a substantive card-contract/integrity failure terminal for that card/telling-set completion (§4.9) — the **telling-completion checkpoint is not appended**, so telling-level semantic fan-out for that reading **remains blocked** by this gate. This block is keyed to the **telling-set completion-operation** state, **not** to the parent pass lifecycle: the immutable parent reading and its already-`completed` pass lifecycle are unchanged, and the block is lifted only when the complete valid card set (or the zero-telling sentinel) legitimately exists. Throughout, the incomplete/failed card-stage artifacts remain **discoverable for governance** (§5A.2), never treated as semantic evidence.

## 5A.2 Never blocked by the semantic gate, when properly authorized (SAFETY/GOVERNANCE discovery)

The semantic-eligibility gate **never blocks** the following from discovering and governing an artifact (including a preparation envelope, a partial card, an integrity-failed card, a blocked legacy entry, a checkpoint, or an index entry), subject to the artifact's own §7Q authorization and §25 identity/security authorization:

- §7Q dependency and derivative discovery;
- privacy / restriction / deletion / influence-removal enforcement;
- integrity checking;
- duplicate detection;
- crash recovery;
- security inspection;
- schema validation;
- operational audit;
- quarantine containment;
- locating and governing partial, blocked, or integrity-failed artifacts.

A safety or governance mechanism **may discover and govern an artifact without treating it as semantic evidence or a canonical telling.** Discovery-for-governance is not semantic use.

## 5A.3 Reading independence, idempotent resume, logging

**Reading independence vs the valid-manifest gate.** **Root ingestion remains independent and non-blocking** (`append_root()` is durable upstream and never waits on the mouth — Master §7G-A). The **reading commit, however, is gated by the valid-manifest protocol** — it waits for semantic acceptance, mechanical assembly, the pre-manifest correspondence validation, and a complete valid manifest (§4.3). What the reading commit **does not** wait for is later telling-card completion or telling-specific semantic fan-out: once the reading is committed, reading-level downstream work that does not depend on telling-level cards is not blocked, and only the **telling-specific semantic fan-out** waits for the complete valid card set (§5A.1). A crash before the reading is durable is an RC-4 abandon-and-replace (§4.4); a crash after the reading is durable recovers the committed reading's missing cards from its manifest (case C).

**Idempotent resume.** Once recovery completes the valid card set and appends the completion checkpoint (or the zero-telling sentinel is present), telling-level semantic fan-out **resumes idempotently** — consumers proceed exactly once against the now-complete set, with their own duplicate prevention intact; wait-then-resume produces no double work and no double evidence.

**Logging.** Every **block** (a semantic operation refused because the gate is not satisfied), every **recovery** step, every **no-op** (idempotent skip), and every **resumed fan-out** is recorded under §0B one-operation/one-log, no-double-evidence (§10), with structural identifiers and result states only — never the private payload (§9).

# 6. CROSS-COMPONENT TELLING-ID CONTRACTS

*(Preserved from v1.1 in substance; consumer references aligned with the §5A semantic-vs-safety distinction.)* A2 unlocks exactly one thing for downstream components: a stable `telling_id` they may point at — and, per §5A.1, only for **semantic** use once the telling set is complete and valid. This section defines only that identity interface; it does not complete any consumer's own schema (B2/B3/B4/B5 and the respective A-policies).

**Universal rule for every consumer below:** a downstream reference points to the stable `telling_id` and, through it, preserves access to the parent reading and the supporting roots. **Links are never copies.** A link never copies or silently rewrites the original telling; the telling stays where it is and the consumer holds a reference plus its own link metadata (what it connects, why-by-pointer, who established it, certainty, timestamp — per Master §7L line 2226 where applicable). **No consumer may make semantic use of a telling that has not passed the §5A.1 gate; §7Q/security discovery of any artifact is governed separately by §5A.2.**

- **C-7K Story Layer (owner).** Holds the first-class telling cards. Organizes tellings by perspective/theme/time using `telling_id` as the stable handle. Connections across time are explicit navigational links keyed on shared attributes (same `perspective_owner`/`whose`, same theme, same thread/source, overlapping time, shared root ids) — navigational links, not logical merges, always labeled with their basis and always pointing back to supporting roots and readings (Master §7K line 2190).
- **C-7L Person-Boxes.** Links story tellings in which a person appears in any perspective role, by `telling_id`. The link record preserves what/why/who-established/certainty/timestamp. Linking a telling never copies it and never promotes it to a fact about the person. *(Person-Box evidence/merge policy remains A5; link-schema mechanics remain B4.)*
- **C-7J Clash Handling (where telling-level links are used).** A clash record points to the exact readings and roots involved (Master §7J line 2160); where a clash is between *tellings* specifically, it may additionally point to the `telling_id`s involved. The same clash never produces two records (later detection appends a detection-history event). Clashes are surfaced, never resolved; pointing at conflicting tellings never selects a winner. *(Clash-record schema specifics remain B2; clash-response downstream policy remains A9.)*
- **C-7M Computed View.** Lists tellings among its source-object types (Master §7M line 2262), referenced by `telling_id`, linked never copied. Seven-factor ordering and internal-only rules unchanged; telling identity gives a stable handle, nothing more. *(Snapshot-schema structure remains B5; relevance mode remains A4.)*
- **C-7D Living State Web.** Story tellings are eligible source objects for state nodes (Master §7D lines 645, 655), referenced by `telling_id`. The grounding rule still requires immediate source-object ids plus direct root ids and a fully inspectable chain; circular support remains forbidden; Story-Layer firmness rules apply when tellings are used as evidence. Telling identity supplies the stable source-object id; it supplies no evidential weight. *(Living State Web evidence/currentness/domain policy remains A6.)*

# 10. §0B LIVING-RECORD LOGGING — NO CONTENT LEAK

Per §0B (Master lines 233–259): every real operation creates one append-only living-record log; **one operation, one log**; a log proves an operation occurred and **never becomes double evidence** that the underlying telling is true; no silent operation. **Logs carry structural identifiers, digests, result states, and failure classes — never the private telling payload** (§9). Logs are subject to §7Q access/authorization and §25 identity/security authorization where applicable. Component-specific logging:

- **Semantic acceptance result** — one log recording the acceptance-check outcome, preserving the **validator version**, the **criterion outcomes**, **pass/failure reasoning or grounded-reasoning references**, and, when rejected, the **specific failure reason** — by reason type, before any durable preparation, and **without copying the private proposal payload** into the log (§9).
- **Mechanical assembly result** — one log when the DUMB layer attaches and validates the system fields (shape/uniqueness/provenance-consistency/referential-integrity, including `supporting_root_ids` resolution and the exact legacy-value copy), by identifier/result state — distinct from semantic acceptance, never semantic content.
- **Pre-manifest correspondence-check result** (§3 rule 8) — one log recording that the deterministic embedded-vs-first-class correspondence validation passed, or that it failed with a **distinct structural/correspondence failure reason** — by identifier/result state, never the payload; on failure the manifest never becomes valid and nothing commits.
- **Manifest-validity event** (§4.2 contract) — **the single log** for manifest validity, recorded once when the manifest-validity event is appended (carrying `prepared_at`): the validity-event operation identity (derived from `preparation_id`), `preparation_id`, `preparation_idempotency_key`, assigned `reading_id`, expected telling count (including zero), the assigned `telling_id`s (as identifiers), and the integrity digests — **never the reading or telling content itself.** A repeated append is an idempotent no-op (no duplicate log).
- **Preparation terminal result** — when a preparation closes by cause (§4.4.1A / §4.9): exactly one terminal event — **`abandoned`** (crash-interrupted, with `abandonment_reason`) or **`failed`** (with `failure_stage` + `is_technical_and_potentially_retryable`). The **`abandoned` pass-lifecycle event is the single record for an abandonment** — there is **no separate "abandonment transition" log.** Only the `abandoned` crash-interrupted case is followed by a **later, genuinely separate** replacement-pass-creation log carrying `recovery_of_pass_id` lineage. By identifier/digest/result state only.
- **Telling-card commit event** (§4.4.1 contract) — **the single log** for that card's commitment, recorded once when the card-commit event (carrying `committed_at`, idempotency-keyed to the card's `telling_idempotency_key`) is appended: `telling_id`, `telling_idempotency_key`, that a first-class card was committed from the manifest. A repeated append is an idempotent no-op (no duplicate commit-event log).
- **Telling-completion checkpoint / zero-telling sentinel** (§4.4.1 contract) — **the single log** for telling-set completion, carrying the completion-operation identity, parent reading ID, expected telling count, the verified inventory or its digest, and the result type (`complete-card-set` or `legitimate-zero-telling`). Retries append no duplicate.
- **Parent-reading and root linkage** — recorded with commit: `parent_reading_id` and `supporting_root_ids` (identifiers), so the grounding chain is inspectable without content.
- **Materialization from an embedded entry** — one log per materialized card, recording that it was materialized and from which embedded entry (identifiers).
- **Blocked legacy-materialization** — one log per blocked legacy entry: the exact missing mandatory requirements (as field names/result state) and that the original embedded entry is preserved unchanged (§4A.2) — no payload leak.
- **Duplicate-prevention result** — recorded when a write is skipped because the `telling_idempotency_key` already exists.
- **Integrity-check result** — for each card's correspondence/integrity check (§3 rule 5): pass, or an integrity failure with what mismatched (by field/digest), without rewriting any immutable source and without leaking content.
- **Partial-completion recovery** — what recovery did (which keys it committed from the envelope, which it skipped, when it appended the completion checkpoint or zero-telling sentinel), by identifier.
- **Semantic-gate block / resume** (§5A) — each semantic operation **blocked** by the gate and each **resumed** fan-out after the set becomes complete; each an inspectable event that adds no certainty to any telling's correctness.
- **Safety-discovery events** (§5A.2) — §7Q discovery, integrity checks, security inspection, and audit that locate/govern an artifact are recorded as their own operations (identifier/result state), distinct from semantic use.
- **Retry or terminal failure** — distinct reason types per the §4.9 taxonomy.
- **Telling-level link creation** — one log per downstream link established to a `telling_id` (Person-Box link, theme link, clash telling-pointer, Living-State-Web source reference, Computed-View reference): what was linked, by which component, its provenance pointer.
- **Rejected or blocked downstream use** — recorded when a link or use is blocked (a quarantined telling refused as production material; a §7Q visibility block; a §5A.1 semantic-gate block; an integrity-failed card refused as canonical).
- **Schema/version provenance** — every card, every envelope, and every telling-level link carries the `schema_version` / producer-version provenance tracing it to the exact producing logic and model.

**No-double-evidence reminder, binding on all of the above:** repeated logs about a telling — its preparation, commit, integrity check, use, linking, blocking, resuming, or governance — never add certainty that the telling is *correct*. One underlying telling must not become several independent votes for itself (Master §0B line 243). Logging records that operations happened; it never strengthens the interpretation, and it never leaks the content.
